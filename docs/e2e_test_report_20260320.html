원인 분석:

  • Binary tasks (ctr, cvr): 어댑터에서 랜덤 생성한 라벨 → Teacher도 학습 불가 → Student도 AUC=0.50
  • Regression tasks (ltv, spending_bucket): 로그 변환 적용했으나 원본 데이터와 라벨 간 상관관계 부족
  • Multiclass tasks: 원본 피처에서 파생한 라벨이라 상관관계 존재 → 학습 가능 (accuracy 92~100%)

10. 시사점 및 향후 과제

9.1 파이프라인 안정성

  • Python 3.9/3.10 호환성: SageMaker 이미지마다 Python 버전이 다름. PEP 604 문법은 프로젝트 전체에서 제거 필요. CI에 Python 3.9 lint 추가 권장.
  • source_dir 패키징: SageMaker SDK의 Estimator와 Processor가 소스 코드 전달 방식이 다름. 통일된 패키징 유틸리티 필요.
  • 모델 checkpoint 직렬화: PLEConfig를 pickle이 아닌 dict로 저장하므로, 재구성 로직이 양쪽에서 일관되어야 함.

9.2 모델 품질

  • 합성 라벨 한계: Binary task AUC=0.50은 데이터 문제. 실제 금융 데이터에서는 Teacher도 학습 가능하므로 Fidelity도 개선될 것.
  • Multiclass/Regression: 원본 피처와 상관관계가 있는 라벨은 학습 성공 (accuracy 92~100%, R²=0.33~0.66). 파이프라인 자체는 정상.
  • ealtman2019 데이터셋 (2,400만 건): 거래 시계열이 있어 TDA/HMM/Mamba 피처 생성기까지 테스트 가능. GPU 할당량 승인 후 진행.

9.3 서빙 성능

  • Warm latency 9.9ms: 금융권 실시간 추천 기준(200ms)의 1/20 수준. 충분한 여유.
  • Cold start 6.3초: 16개 모델을 S3에서 다운로드하는 시간. Provisioned Concurrency(월 ~$20) 또는 Lambda SnapStart로 해결.
  • Lambda Layer 54MB: lightgbm + scipy + numpy + libgomp. 250MB 제한의 22% 사용.
