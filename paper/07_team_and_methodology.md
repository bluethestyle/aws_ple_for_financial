# Team Structure & AI-Augmented Development Methodology

## 1. 팀 구성과 제약 인식

### 팀 규모
- PM 1명 + 팀원 2명 (총 3명)
- 어떤 회사든 인력은 항상 부족하고 생산성은 늘려야 하는 상황
- 처음부터 인력 부족을 인지하고, 할 수 있는 가장 베스트 케이스를 추진
- 하드웨어/자원 한계에서 효율을 뽑아냈듯, 소규모 팀에서 할 수 있는 방식으로 적응

### PM의 도메인 경험이 아키텍처에 녹아든 구조
PM의 경력이 파이프라인의 각 요소에 직접 반영:

| 경력 영역 | 파이프라인 반영 |
|-----------|--------------|
| FRM (시장/신용 리스크 관리) | Evidential uncertainty, 리스크 기반 추천 제약 |
| 컴플라이언스 | 금감원 가이드라인 매핑, EU AI Act 대응 설계 |
| 신용분석 | 고객 세그먼트 설계, 경제학 피처 (항상소득/한계효용) |
| 디지털 신사업 기획 | end-to-end 파이프라인 아키텍처, 서버리스 서빙 설계 |
| 마이데이터 허가 획득 | 데이터 거버넌스, 개인정보 비식별 처리 구조 |
| 빅데이터 플랫폼 구축/운영 | DuckDB/cuDF 백엔드 선택, Spark Ingestion 확장 설계 |
| 데이터 분석과제 수행 | 다학제 피처 설계 (TDA, HMM, 경제학 지표 등) |
| 추천 알고리즘 관리 | ALS → PLE 전환 판단, 증류 + Lambda 서빙 구조 |

이 프로젝트가 단순한 기술 데모가 아니라 **금융 현장의 실제 요구사항을 반영한 설계**인 이유:
현장에서 규제 대응, 플랫폼 운영, 추천 시스템 관리를 직접 경험한 사람이
각 단계에서 "현실에서 이게 되는가"를 검증하며 설계했기 때문.

### 핵심 전략
1. **설계 선행**: 전체 아키텍처를 견고하게 디자인 → 작업 변수 최소화
2. **스크럼/스프린트**: 빠른 의사결정 사이클
3. **AI 에이전트 적극 활용**: 3명의 인간이 각각 AI 팀을 이끄는 구조

## 2. 개발 프로세스

### Phase A: 아이디에이션 & 설계 (Gemini)
```
PM + Gemini
  ├── 브레인스토밍: 추천 아키텍처 후보 탐색
  ├── 컨셉 정리: Black-Litterman → MTL → PLE+adaTT 의사결정 과정
  ├── 다학제 피처 설계: 경제학/위상수학/금융공학 관점 정리
  └── 초기 아키텍처 문서화
```

### Phase B: 기술 검증 & 구체화 (Opus)
```
PM + Opus
  ├── 구현 가능성 검증: 각 expert의 기술적 실현 가능성
  ├── 아키텍처 살 붙이기: config 구조, 데이터 흐름, 인터페이스 설계
  ├── 기술 참조 문서 작성
  └── 트레이드오프 분석: 파라미터 수 vs 표현력, 서빙 비용 등
```

### Phase C: 개발 환경 정비 & 설계서 작성 (Cursor)
```
팀 전체 + Cursor (GitHub 기반)
  ├── Git 워크플로우 정비
  ├── 6개 설계 문서 초기 작성 (00~09)
  ├── 코드 규약 정립 (CLAUDE.md)
  └── 프로젝트 스캐폴딩
```

### Phase D: 병렬 구현 (AI 에이전트 팀 운영)
```
PM (팀장 역할)          팀원 A (팀장 역할)       팀원 B (팀장 역할)
  │                       │                       │
  ├── Opus (설계/검증)     ├── Opus (설계/검증)     ├── Opus (설계/검증)
  ├── Sonnet ×N (구현)    ├── Sonnet ×N (구현)    ├── Sonnet ×N (구현)
  │                       │                       │
  ▼                       ▼                       ▼
모델 레이어              데이터 레이어             서빙/설명 레이어
(PLE, experts,          (adapter, generators,   (distillation,
 adaTT, towers)          normalizer, pipeline)   reason, Lambda)
```

각 인간 팀원이 AI 에이전트들의 **팀장** 역할:
- 역할 배분: 레이어별 구현을 Opus/Sonnet에 분배
- 품질 관리: 코드 리뷰, 인터페이스 일관성 검증
- 맥락 유지: 에이전트 간 설계 방향성 정렬

### AI 에이전트 도구별 역할 분담
각 도구의 강점을 파악하고 용도를 명확히 분리:

| 도구 | 강점 | 활용 용도 |
|------|------|----------|
| **Gemini** | 광범위한 지식, 브레인스토밍 | 아이디에이션, 컨셉 탐색, 다학제 참고문헌 조사 |
| **Opus** | 깊은 추론, 아키텍처 설계 | 기술 검증, 설계 구체화, 트레이드오프 분석, 복잡한 버그 진단 |
| **Sonnet** | 빠른 코드 생성 | 구현, 리팩토링, 테스트 작성 |
| **Cursor** | GitHub 통합, 코드베이스 이해 | 코드 네비게이션, 파일 간 참조, Git 워크플로우 |
| **Claude Code Extension** | VS Code 통합, 실시간 편집 | 디버깅, 실시간 코드 수정, 로컬 실행/모니터링 |
| **Teams 기능** | 멀티 에이전트 협업 | 병렬 작업 분배, 에이전트 간 작업 조율 |

핵심: 하나의 AI를 만능으로 쓰지 않고, **각 모델이 잘하는 부분에 특화**시켜
마치 이종 expert PLE처럼 AI 도구들도 전문화 + 조합으로 운영.
프로젝트 아키텍처 철학(이종 전문가 조합)이 개발 방법론에도 일관되게 적용된 셈.

## 3. 품질 유지 전략

### 가드레일 (CLAUDE.md)
전반적인 코드 규약과 설계 방향성을 CLAUDE.md에 정리:
- Config-driven 원칙 (하드코딩 금지)
- 관심사 분리 (Adapter/Pipeline/Trainer)
- 데이터 리키지 방지 규칙
- 커밋 전 4단계 검수 (컴파일/인터페이스/하드코딩/관심사)
- 서브에이전트 사용 규칙

이 가드레일이 여러 AI 에이전트가 병렬로 작업해도
코드 품질과 아키텍처 일관성을 유지하는 핵심 장치.

### 맥락 유지 전략 (컨텍스트 윈도우 극복)
AI 에이전트의 컨텍스트 윈도우 제약을 극복하기 위해:
- **메모리 뱅크**: 세션 간 핵심 결정사항/진행상황을 파일로 영속화
- **진행 과정 문서화**: 각 작업의 상태/결과를 별도 문서로 기록
- **설계 문서 참조**: 에이전트가 항상 설계서를 읽고 맥락을 복구
- **CLAUDE.md**: 프로젝트 규칙이 매 세션 자동 로드

```
세션 1                    세션 2                    세션 3
  │                         │                         │
  ├── 작업 수행              ├── 메모리 뱅크 로드         ├── 메모리 뱅크 로드
  ├── 결과 기록              ├── 이전 맥락 복구          ├── 이전 맥락 복구
  └── 메모리 뱅크 저장        ├── 작업 이어서 수행        ├── 작업 이어서 수행
                            └── 메모리 뱅크 갱신        └── 메모리 뱅크 갱신
```

## 4. 설계 변경 관리

중간에 지속적인 설계 변경이 있었지만, **방향성**은 변하지 않음:

| 변경 사항 | 이유 | 방향성 유지 |
|-----------|------|-----------|
| GMM 서브헤드 → 태스크 그룹 | 복잡도 폭발 | 멀티태스크 전문화는 유지 |
| Black-Litterman → PLE+adaTT | 비즈니스 해석 불가 | 전문가 의견 통합 개념은 유지 |
| pandas → DuckDB/cuDF | 메모리 한계 | 데이터 처리 효율화 방향 유지 |
| Docker 로컬모드 → Python 직접 | WSL GPU 문제 | 검증 목적은 유지 |
| 합성 데이터 도입 | 실데이터 AUC ceiling | ablation 의미있는 비교 목적 유지 |

견고한 초기 설계 덕분에, 변경이 "구현 방식"에 한정되고
"아키텍처 철학"을 흔들지 않았음.

## 5. 논문에서의 의미

### AI-Augmented Development as Methodology
이 프로젝트 자체가 "소규모 팀이 AI 에이전트를 활용하여
복잡한 ML 시스템을 구축할 수 있는가"에 대한 사례 연구.

- 3명이 6개월 이내에 13-task, 7-expert, 23-scenario ablation 시스템 구축
- 기존이라면 ML 엔지니어 5-10명 규모의 작업
- 핵심 성공 요인: 견고한 설계 + 가드레일 + AI 에이전트 병렬 운용

### 재현 가능성
- 설계 문서 + CLAUDE.md + config 파일이 공개되면
  다른 소규모 팀도 동일한 방법론으로 유사 시스템 구축 가능
- AI 에이전트 활용 방법론 자체가 논문의 부가적 기여(contribution)
