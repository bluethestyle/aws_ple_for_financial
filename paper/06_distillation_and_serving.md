# Knowledge Distillation & Serving Pipeline

## 1. 증류 전략

### 배경: 추론 속도 문제
이종 expert PLE+adaTT는 표현력은 높지만 추론에 GPU가 필요 (~100ms).
금융사 현실: GPU가 달린 추천 서버가 존재하지 않는 경우가 대부분.
따라서 CPU로 실시간 서빙 가능한 경량 모델로의 증류가 필수.

### Teacher-Student 구조
- Teacher: PLE 18-task, 7 experts, 316 features (GPU, 주 1회 재학습)
- Student: LGBM × 18 (태스크 헤드별 독립 LGBM, CPU, 일 1회 증류)
- 증류 방식: teacher의 soft label(확률 출력)을 각 LGBM이 독립 학습

```
선생 (PLE+adaTT, GPU)              학생 (LGBM × 18, CPU)
  ├── 주기: 주 1회 재학습              ├── 주기: 일 1회 증류
  ├── 환경: SageMaker GPU             ├── 환경: Lambda CPU
  ├── 추론: ~100ms                    ├── 추론: ~5ms
  └── 역할: 최고 성능 + soft label     └── 역할: 실시간 서빙
```

### 태스크 헤드별 독립 증류의 장점
- 특정 태스크만 성능 저하 시 해당 LGBM만 재증류 (전체 재학습 불필요)
- 금감원 모형 검증 대상이 "LGBM 18개" → 개별 설명 가능성 높음
- 선생 모델은 "학습 인프라"로 분류 → 검증 부담 낮음
- 선생이 학습 중이어도 학생은 이전 버전으로 정상 서빙 → 무중단 운영

### 선생/학생 주기 분리
- 금융 데이터 변동이 느림 (일/주 단위) → 선생 주 1회면 충분
- 학생은 최신 soft label로 일 1회 갱신 → 항상 최신 상태 유지
- 선생 모델 장애 시에도 학생은 독립 운영 가능 (resilience)

### IG 기반 피처 선택
- Integrated Gradients로 teacher 모델의 피처 중요도 계산
- 상위 k개 피처만 student에 전달 → 입력 차원 축소
- ablation 결과와 연계: 기여도 높은 feature group의 피처 우선 보존

### 증류 품질 메트릭
- teacher 대비 AUC drop 최소화 (목표: 2-5%p 이내)
- 태스크별 독립 평가 (18개 태스크 각각)
- 증류 후에도 설명 가능성 유지 (LGBM feature importance + 피처 역매핑)

## 2. 추천사유 생성

### 멀티 에이전트 추천사유 생성 파이프라인
단일 LLM 호출이 아닌, 역할 분리된 3-agent 파이프라인:

```
[모델 추론 완료]
      ↓
┌─────────────────────────────────────────┐
│ Agent 1: Feature Selector (피처 선별)     │
│  - IG attribution 상위 피처 선택           │
│  - 추론에 사용된 피처 + 설명력 관점에서     │
│    추가로 중요한 피처를 보충 선별           │
│  - 고객 설득에 효과적인 피처 조합 구성      │
└─────────────────┬───────────────────────┘
                  ↓
┌─────────────────────────────────────────┐
│ Agent 2: Reason Generator (사유 생성)     │
│  - 선별된 피처를 비즈니스 맥락으로 역매핑    │
│  - 고객에게 자연스러운 문장으로 생성         │
│  - 금융 도메인 톤&매너 준수                │
└─────────────────┬───────────────────────┘
                  ↓
┌─────────────────────────────────────────┐
│ Agent 3: Safety Gate (안전성 검증)         │
│  - 위험성 점검: 부적절한 투자 권유 등       │
│  - 규제 위반 점검: 금소법, 적합성 원칙      │
│  - 할루시네이션 점검: 없는 사실 언급 여부    │
│  - 통과 시 → 서빙, 불통과 시 → 재생성/폴백  │
└─────────────────┬───────────────────────┘
                  ↓
[고객/행원에게 전달]
```

이 구조의 이점:
- 역할 분리: 각 agent가 한 가지에 집중 → 품질 향상
- 독립 개선: 사유 생성 품질과 안전성 검증을 독립적으로 튜닝 가능
- 감사 추적: 각 agent의 입출력이 로깅 → 사후 추적 가능
- 폴백: Safety Gate 불통과 시 템플릿 기반 안전한 사유로 자동 전환

### 예시
```
Input: customer_id=12345, 추천상품=투자펀드
Attribution: synth_monthly_spend(0.35), income(0.28), prod_checking(0.20)

Output: "고객님의 최근 3개월 월평균 지출 수준(상위 15%)과
        소득 수준, 입출금 계좌 보유 패턴을 종합적으로 분석하여
        투자 펀드 상품을 추천드립니다."
```

### 규제 충족
- 금감원: "왜 이 상품을 추천했는가" 설명 의무
- EU AI Act Article 13: 고위험 AI 시스템 투명성 요건
- 개별 추천 근거 로깅 → 사후 감사 추적 가능

## 3. Lambda 서빙 아키텍처

### 구조
```
Client → API Gateway → Lambda → Response
                        ├── LGBM 모델 로드 (수 MB)
                        ├── 추론 (< 10ms)
                        ├── Attribution 계산 (< 50ms)
                        ├── 추천사유 생성 (< 5ms)
                        └── 총 latency < 100ms
```

### 모델 업데이트
- 새 모델 학습 완료 → S3 업로드 → Lambda 버전 교체
- Blue/Green 배포: 이전 버전 유지, 문제 시 즉시 롤백
- 모형 검증 통과 전까지 canary 배포 (트래픽 5%만 새 모델)

### 모니터링
- CloudWatch: 추론 latency, 에러율
- 태스크별 prediction 분포 drift 감지
- 주간 AUC 재평가 (ground truth 수집 후)
