# Regulatory Compliance & Explainability

## 1. 규제 환경 매핑

### 금감원 「금융분야 AI 가이드라인」 + 「모형 리스크 관리 지침」

| 금감원 요건 | 프로젝트 대응 |
|------------|-------------|
| 설명 가능성 | IG 기반 추천사유 생성, expert별 기여도 분해 |
| 공정성 | ablation으로 특정 feature group(demographics) 제거 시 영향 분석 |
| 모형 검증 | ablation 결과 = 검증 문서 핵심 근거 |
| 모니터링 | 태스크별 AUC 분리 추적 (18개 독립 메트릭) |
| 감사 추적 | SHAP/IG attribution 로깅 → 개별 추천 근거 소급 가능 |
| 대안 모델 폴백 | 증류 LGBM이 teacher 이상 시 자동 전환 가능 |

### EU AI Act Article 13 (투명성 의무)

- High-risk AI system (금융 신용/추천): 설명 의무 적용
- 추천사유 생성이 단순 부가 기능이 아니라 규제 필수 요건
- IG attribution → 자연어 템플릿 → 고객 대면 설명 가능

## 2. 금감원 관점: 성능보다 거버넌스

금감원은 "AUC가 얼마냐"를 묻지 않는다. 질문하는 것:
- "이 모델이 왜 이런 결정을 내렸는지 설명할 수 있는가?"
- "문제 생겼을 때 즉시 원인 파악하고 롤백할 수 있는가?"
- "특정 고객군에 차별적 결과가 발생하지 않는가?"
- "모델 성능이 저하되었을 때 감지할 수 있는가?"

## 3. 설명 가능성 구현 전략

### Expert-level Attribution
- 각 expert의 output이 최종 prediction에 얼마나 기여했는지 분해
- Ablation 결과로 expert별 marginal contribution 정량화

### Feature-level Attribution
- Integrated Gradients (IG)로 피처별 기여도 계산
- 상위 3-5개 피처를 자연어 템플릿에 매핑
- 예: "고객님의 최근 3개월 거래 빈도(상위 15%)와 예금 보유 패턴을 기반으로 투자 상품을 추천드립니다"

### Task-level Monitoring
- 18개 태스크별 독립 AUC 추적
- 특정 태스크 성능 급락 시 경보 → 해당 태스크만 재학습 가능
- adaTT affinity matrix 변화 추적 → 태스크 간 관계 변동 감지

## 4. 파이프라인 내장 모니터링/리포팅 요소

AI 기본법 및 금감원 가이드라인 방향성에 부합하도록,
규제 대응 요소들을 사후 대응이 아니라 **파이프라인 자체에 내장**:

### 감사 추적 (Audit Trail)
- HMAC + 해시 체인 불변 로그: 추천 이력 위변조 불가
- 7개 감사 테이블: 킬스위치, 동의, 프로파일링, AI거부, 인시던트, 증류, 임베딩
- 일별 JSONL + DynamoDB 이중 저장

### 공정성 모니터링 (Fairness)
- 보호 속성 5개에 대해 DI (Disparate Impact), SPD, EOD 상시 감시
- 임계값 위반 시 자동 경보 → 모델 재검토 트리거

### 드리프트 감시 (Drift)
- PSI (Population Stability Index)로 입력 분포 변화 감시
- 연속 3일 임계 초과 시 자동 재학습 트리거
- 태스크별 AUC 추적 → 특정 태스크만 성능 저하 시 해당 LGBM만 재증류

### 쏠림 탐지 (Herding)
- HHI/Gini/Entropy로 추천 쏠림 감지
- 특정 상품에 추천이 집중되면 시스템 리스크로 경보
- 금융 특화: 상품 쏠림은 영업 리스크 + 규제 리스크

### Champion-Challenger
- 현행 모델(champion)과 신규 모델(challenger) 병행 운영
- A/B 테스트 프레임워크로 성능 비교 → 금감원 모형 검증 절차와 연계
- 증류 모델 교체 시 자동 비교 리포트 생성

### 동의/거부 관리 (Consent & Opt-out)
- 마케팅 동의 변경 이력 추적 (부여/철회/갱신)
- AI 결정 거부(opt-out) 이력 관리 → 정보주체 권리 행사 대응
- EU AI Act Article 22 자동화된 의사결정 거부권 대응

### 거버넌스 보고서
- 월/분기 자동 생성: 모델 성능, 공정성, 드리프트, 인시던트 종합
- 금감원 제출용 포맷으로 자동 변환 가능

### Human-in-the-Loop (사람 개입 원칙)
금감원은 최종적으로 **사람의 개입에 따른 의사결정**을 요구한다.
모든 것이 자동화되어 있더라도, 최종 검수는 사람이 수행:

- **추천사유 샘플링 검수**: 전수 조사는 불가하더라도, 주기적으로
  추천사유 생성 결과를 샘플링하여 품질/적합성/규제준수 여부 검수
- **모델 교체 승인**: Champion-Challenger 비교 결과를 사람이 확인 후 승인
- **인시던트 에스컬레이션**: 자동 탐지된 이상 징후를 사람이 판단하여 조치
- **공정성 리뷰**: 공정성 모니터링 결과를 주기적으로 사람이 리뷰

이는 EU AI Act의 "human oversight" 요건과도 부합하며,
파이프라인이 "사람을 대체"하는 것이 아니라
"사람의 의사결정을 지원"하는 도구임을 명확히 하는 설계.

## 5. 논문 차별점

기존 금융 추천 논문들의 한계:
- 모델 성능(AUC/F1)만 보고 → 규제 대응 미언급
- 설명 가능성은 SHAP 한 줄 → 실제 서빙 파이프라인 없음
- 배포/운영 아키텍처 부재 → 학술 실험으로 끝남

본 논문의 차별점:
- 학습 → 증류 → 설명 가능한 서빙까지 end-to-end
- ablation으로 각 컴포넌트의 기여도를 정량적으로 증명
- 금감원 + EU AI Act 규제 요건과 아키텍처의 명시적 매핑
- 서버리스 배포로 금융사 현실에 맞는 TCO 최적화
